# 协作空间邀请码技术方案（闭环版）

本文档描述 ArcPilot 协作空间邀请的**推荐技术路线**：在保留现有「表驱动邀请」能力的前提下，引入**带版本签名的无状态令牌**作为可选路径，形成可演进、可审计、可撤回的**混合模型**。文中与实现对齐的表名/模型为：`collab_spaces`（`CollabSpace`）、`space_invites`（`SpaceInvite`）、`collab_space_members`（`CollabSpaceMember`）。

---

## 1. 目标与原则

| 目标 | 说明 |
|------|------|
| 防伪造 | 邀请载体不可被篡改角色、空间标识、过期时间等关键字段。 |
| 可撤回 | 支持「整空间一键作废历史链接」；可选支持「单条邀请作废」。 |
| 可审计 | 能回答谁发起邀请、何时被使用（至少满足一般合规与排障）。 |
| 成本可控 | 避免无差别地为每一次「复制链接」堆积海量行；热点路径可水平扩展。 |

**原则**：不追求「零数据库行」的极端形态，而追求**能力与存储的合理平衡**。

---

## 2. 混合模型总览

### 2.1 两种邀请载体（并存）

1. **表驱动邀请（当前已实现）**  
   - 每条邀请对应 `space_invites` 一行：`invite_code`、`role`、`expires_at`、`status`、`inviter_id` 等。  
   - **适合**：需要**单条撤销**、**强审计**、**邮件/定向邀请**、**单次或有限次使用**（通过 `status` 流转实现）。

2. **无状态签名令牌（推荐逐步引入）**  
   - Payload 经 **HMAC-SHA256** 签名后编码为单一字符串（如 Base58）；服务端**不依赖**「先插库再校验」即可完成验证。  
   - **空间级版本号**：在 `collab_spaces` 增加 `invite_version`（整数，默认 `1`）。管理员执行「重置本空间邀请」时执行 `invite_version += 1`，使此前发出的无状态码中 `Ver` 低于当前版本的一律失效。  
   - **适合**：高频「分享链接」、希望**减少表写入**、接受「按空间一键清空」而非「按链接逐条撤销」。

### 2.2 校验入口统一

用户提交的「邀请码」在 API 层按序尝试解析：

1. **无状态令牌**：解析 → 验签 → 校验 `Exp` → 拉取空间 → 校验 `payload.Ver == space.invite_version`。  
2. **若失败**：再按**纯随机短码**查 `space_invites`（现有逻辑），校验 `status`、`expires_at`。

这样旧数据与新模式可平滑共存。

---

## 3. 无状态令牌规范（建议）

### 3.1 Payload 字段（示例）

使用 **JSON** 或 **固定顺序二进制** 序列化后作为 HMAC 输入（实现时二选一并版本化 `typ` 字段）。

| 字段 | 含义 |
|------|------|
| `typ` | 令牌类型，如 `cs_inv_v1`，便于未来换算法。 |
| `sid` | 空间对外 ID：`CollabSpace.public_id`（UUID），**禁止**仅依赖内部自增 `id` 对外暴露。 |
| `role` | `editor` / `viewer`（与现有 `SPACE_VALID_ROLES` 一致，不含 `owner`）。 |
| `exp` | Unix 时间戳（秒），过期时间。 |
| `ver` | 签发时自 `collab_spaces.invite_version` 读取的快照，**必须**与接受时库中值一致。 |
| `jti` | 可选；若需「单条撤销」且无大表，可配合撤销集合使用（见 §5）。 |

**签名**：`sig = HMAC-SHA256(secret, canonical_payload)`。  
**传输**：`token = Base58Encode( payload_bytes || sig )`（或 Base58 包裹 `typ`+payload+sig 的标准结构）。

> 说明：流程是 **验签 + 解析**，不是对称「解密」；对外文档避免误用「解密」一词。

### 3.2 密钥与轮换

- 使用独立配置项（如 `COLLAB_INVITE_HMAC_SECRET`），与 `SECRET_KEY` 分离，降低误用面。  
- **轮换**：可维护 `kid` 或 `typ` 版本，短期内允许多把密钥验签；旧码在宽限期后自然过期。

### 3.3 时钟与体验

- `exp` 校验时允许 **±120s** 时钟偏差，减少边缘失败。  
- 限流与风控见 §5。

## 4. 端到端业务流程

### 4.1 生成无状态邀请（Owner/Editor 有权限时）

```text
1. 鉴权：当前用户对目标 space 具备邀请权限（与现有 SPACE_INVITE_ROLES 一致）。
2. 读库：SELECT invite_version, public_id FROM collab_spaces WHERE ...
3. 组装 Payload：sid=public_id, role, exp, ver=invite_version, typ=...
4. 计算 HMAC，Base58 编码得到 token。
5. 可选：写审计日志（异步），不强制写 space_invites 行。
6. 返回给前端展示/复制；支持 ?code= 深链。
```

### 4.2 接受邀请（与现有 `/invites/accept` 对齐）

```text
1. 限流：按 IP + 可选按 user_id（见 §5）。
2. 尝试无状态解析与验签；成功则解析出 sid → 映射内部 space_id。
3. 校验 ver == collab_spaces.invite_version；校验 exp；校验 role 合法。
4. 幂等：若已是成员，可返回成功或明确「已是成员」（与现网约定一致）。
5. 写入/更新 collab_space_members；无状态路径可选写 invite_redemptions 窄表（§6）。
6. 若第 2 步失败：走现有 space_invites 查表 + status + 过期判断 + accept 更新。
```

### 4.3 整空间「重置邀请」

```text
1. 仅 Owner（或策略规定的管理员）：UPDATE collab_spaces SET invite_version = invite_version + 1。
2. 可选：将本空间所有 pending 的 space_invites 标记为 cancelled（表驱动路径一并失效）。
3. 审计日志记录操作者与时间。
```

## 5. 已知问题与对策

| 问题 | 说明 | 对策 |
|------|------|------|
| 无状态码无法「只撤销一条」 | `invite_version` 是空间级，+1 会使**所有**旧无状态链接失效。 | 产品明确：默认一键清空；若需单条撤销，使用 **表驱动邀请** 或 Payload 带 **jti** + **撤销表/Redis Set**。 |
| 单次使用 vs 多人加入 | 纯无状态无法区分「码只能用一次」。 | 需要单次则：**写库**（`space_invites`）或接受成功后写 **`jti` 已使用** 记录。 |
| 审计不足 | 完全无状态且不记日志时难以追责。 | 接受成功写 **`invite_redemptions`** 或结构化日志；表驱动邀请保留 `inviter_id`、时间戳。 |
| 限流仅按 IP | NAT 下误伤、共享出口绕过。 | **IP + 用户账号** 组合限流；失败退避；监控异常 IP。 |
| 暴力尝试 | HMAC 验证便宜，但仍可被扫。 | 全局限流、验证码（高危时）、监控告警。 |
| 密钥泄露 | 泄露后可伪造邀请。 | 密钥独立、定期轮换、环境隔离、泄漏应急时轮换 + 提升 `invite_version`。 |

## 6. 数据模型演进建议（与实现对齐）

| 变更 | 用途 |
|------|------|
| `collab_spaces.invite_version` | INT，默认 1；重置邀请时递增。 |
| 保留 `space_invites` | 表驱动邀请、邮件定向、单条状态机。 |
| 可选 `invite_redemptions`（窄表） | `id, space_id, user_id, token_jti_or_hash, accepted_at`；用于无状态路径审计与防重复（若产品需要）。 |

迁移策略：先加列与默认值，再上线无状态签发/校验双轨逻辑，最后按产品开关放量。

## 7. 测试方案

### 7.1 单元测试

- **HMAC**：已知 `secret` 与 payload，签名结果稳定；篡改 payload 任意比特 → 验签失败。  
- **过期**：`exp` 在过去 → 拒绝；边界 ± 时钟容差。  
- **版本**：`ver < DB.invite_version` → 拒绝；相等 → 通过。  
- **角色**：`owner` 或非法字符串 → 拒绝。

### 7.2 API / 集成测试（pytest）

- 创建空间 → 签发无状态码（或直接调路由）→ `accept` → 成员存在且角色正确。  
- `invite_version` +1 后，**旧码** accept 失败；**新码**在版本对齐后可成功。  
- 与 **表驱动** 邀请交叉：同一次 accept 接口，表码与无状态码两条用例。  
- 已是成员再次 accept：行为符合幂等约定。  
- 过期表邀请、`status != pending`：拒绝。

### 7.3 安全与回归

- 伪造随机字符串：应失败且**不泄露**空间是否存在（错误信息宜泛化）。  
- 限流：高频失败触发 429（若实现）。  
- 性能：验签路径压测 QPS 与延迟基线。

### 7.4 前端（可选 E2E）

- 复制邀请码 → 另一账号登录 → 加入成功 → 列表可见空间。  
- 深链 `?code=` 自动填充并提交（若产品支持）。

---

## 8. 与旧版「纯无状态」叙述的差异（文档闭环说明）

早期草案强调「不存库、仅版本号」，在**单条撤销、单次使用、强审计**上存在天然缺口。本方案通过 **混合模型** 与可选 **`jti` + 撤销集合 / 窄表** 补齐闭环，并明确与现有 **`space_invites`** 的关系，避免实现阶段返工。
