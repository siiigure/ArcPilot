# 城乡规划专业知识库（Wiki）— 可执行技术方案

本文档为 **最终执行版**：在「可维护、数据库与性能平衡、简洁可演进」前提下，固定数据流与选型，避免「两处真相」与管线冲突。实现时以本方案为准，与《数据库逻辑模型》《协作空间》中的 `space_id`、权限模型对齐。

---

## 1. 目标与边界

| 维度 | 要求 |
|------|------|
| 单一事实来源 | 文档 **元数据与权限** 在业务库（PostgreSQL）；**正文** 在对象存储；前端不直接信本地 Markdown 仓库为权威。 |
| 多租户 | 所有文档归属 `space_id`；目录树、搜索、详情均按空间与成员权限过滤。 |
| 性能 | 目录树与热点页可缓存；全文检索走服务端索引，不在浏览器加载全量正文做 Fuse。 |
| 演进 | P0 可先上「关键词检索」；向量/RAG 与同一套「发布流水线」挂钩，避免双轨不同步。 |

**不在本文展开**：具体 UI 像素级规范；与社区 `Post` 混排——Wiki 为独立内容类型，仅复用用户与空间体系。

---

## 2. 产品与交互（摘要）

- **布局**：三栏（导航约 260px / 正文最大宽约 800px / 右侧 H2–H3 大纲），风格参考系统级文档站（可读性优先）。
- **导航**：由 **API 返回的目录树** 驱动，**不**依赖磁盘路径或写死 `/standards` 等顶层（分类由表 `wiki_category` 或等价结构配置）。
- **快捷键**：`Cmd/Ctrl + K` 打开搜索面板；结果来自 **搜索 API**（见 §5），非全量 `search-index.json`。
- **规划小工具**：阅读区可嵌入容积率等小组件（前端组件，与正文元数据无关）。

---

## 3. 逻辑架构（闭环）

```
运营/管理员保存文档
    → API 校验 space 与权限
    → 正文写入 S3/MinIO（版本化 key，见 §4）
    → DB 更新元数据（slug、title、version、updated_at、category_id 等）
    → 异步任务：重建 FTS 索引片段 +（可选）向量索引
    → 失效缓存：Redis 目录树 tag + Next 侧 revalidate（见 §6）
```

**读路径**：用户请求 → 鉴权 → 查 DB 元数据 → 若有权限则拉取 S3 正文 → 服务端 MDX 编译渲染 → 返回 HTML/流式页面。

---

## 4. 存储与内容管线（必须二选一的已固定）

### 4.1 元数据（PostgreSQL）

建议最小字段（可与现有表合并命名，但语义一致）：

- `id`, `space_id`, `slug`（空间内唯一）, `title`, `author_id`
- `category_id`（可空，指向配置化分类表）
- `visibility` 或与协作空间角色对齐的枚举/位标志
- `content_version` / `updated_at`
- `storage_key`（S3 对象 key，含版本或内容 hash 便于回滚）

**权限**：与协作空间成员关系一致——「谁能读/写某 space 下的 wiki」全部由 **后端 API** 判定；Next.js 不做文件系统鉴权。

### 4.2 正文（S3 兼容存储）

- 存 **`.md` 或 `.mdx`**；MIME 与编码统一 UTF-8。
- Key 规范示例：`wiki/{space_id}/{doc_id}/{version}.mdx`，避免覆盖写丢历史（版本策略在 DB 记录当前指针）。

### 4.3 编译方式（与「动态正文」一致）

| 场景 | 做法 |
|------|------|
| **在线编辑后阅读（主路径）** | 后端或 BFF 从 S3 拉取正文，使用 **`next-mdx-remote`（或 RSC 兼容的 MDX 序列化方案）** 在 **请求时或短时缓存** 内编译为 React 树。不采用 Contentlayer 对 S3 做构建时扫描。 |
| **可选：仓库内种子文档** | 若需 Git 管理少量内置文档，可用 **Velite** 在 CI 只对 `content/` 生成类型与导航 JSON；与 S3 文档 **来源不同**，须在代码中区分 `source: repo | storage`，避免混为一谈。 |

**结论**：产品级 Wiki 的默认管线是 **S3 + 运行时 MDX**；Velite 仅作辅助，不替代主链路。

## 5. 搜索与 RAG（可分期）

### 5.1 全文检索（P1 必做）

- **推荐**：**Meilisearch**（或 OpenSearch）托管索引字段：`space_id`, `doc_id`, `title`, `body_plain`（由正文抽取的纯文本）, `slug`。
- **备选（极简单机）**：SQLite **FTS5** 仅适用于低并发、可接受单文件瓶颈时；上线前需评估迁移到 Meilisearch 的条件。

**权限闭环**：搜索请求 **必须** 带用户身份/会话；服务端先解析用户可访问的 `space_id` 集合，查询引擎使用 **过滤条件**（如 Meilisearch `filter space_id IN [...]`）或在结果集上二次过滤 `doc_id`，**禁止**仅依赖前端隐藏结果。

### 5.2 语义检索 / RAG（P2 可选）

- 与全文索引共用同一 **「文档已发布」事件**：发布成功后异步 **分块 → 向量化 → 写入向量库**（与现有 RAGFlow/Milvus 等对接）。
- **一致性**：同一 `doc_id` 更新时，应对 FTS 与向量 **同一任务或同一事务消息** 触发重索引；失败重试、幂等（以 `doc_id + version` 为键）。

### 5.3 混合检索（产品成熟阶段）

- 对用户查询同时走 **关键词** 与 **向量**，再融合排序（加权或 RRF）；权限过滤仍在服务端完成。

## 6. 前端路由与缓存

### 6.1 路由

- 采用 **`/[spaceId]/wiki/[...slug]`**（或项目统一的前缀），目录与面包屑来自 **GET 目录树 API**。

### 6.2 Next.js 缓存策略

- **ISR**：可对 `revalidate` 设合理秒数（如 3600）作兜底；**文档更新后** 必须由后端或 BFF 调用 **`revalidatePath` / `revalidateTag`**（与 `doc_id` 或 `space_id` 绑定），保证运营发布后短时间可见。
- **万篇规模**：`generateStaticParams` **不**强求枚举全部 slug；使用 **按需生成**（`dynamicParams: true`）+ 上述按需失效。
- **Redis**：缓存「某 space 的目录树 JSON」，TTL + **在文档/分类变更时主动删除 key**。

## 7. 技术栈建议（与维护风险）

| 层级 | 选型 | 说明 |
|------|------|------|
| Web | Next.js App Router | 页面与 RSC；Wiki 页以服务端拉正文为主。 |
| 正文渲染 | `next-mdx-remote` + 受控组件映射 | 避免 Contentlayer 作为主路径；自定义 MDX 组件需白名单。 |
| API | 与 ArcPilot 现有后端一致（如 Django REST） | 统一鉴权与 `space_id`。 |
| 对象存储 | S3 / MinIO | 与 `.env` 配置一致。 |
| 检索 | Meilisearch（主推）或 SQLite FTS5（过渡） | 权限过滤必做。 |
| 缓存 | Redis | 目录树；可选页面片段缓存。 |

## 8. 交付阶段（可执行）

| 阶段 | 内容 | 交付物 |
|------|------|--------|
| **P0** | Wiki 元数据表 + `space_id` 约束；S3 上传/读取 API；鉴权与可见性规则文档 | Schema 迁移、API 契约、Postman/测试用例 |
| **P1** | 目录树 API；Wiki 详情页（拉 S3 + MDX 渲染）；发布流程写 DB+S3；**Meilisearch 索引与搜索 API** | 可演示的端到端编辑-发布-搜索 |
| **P2** | 异步重索引任务；向量库与 RAG 对齐；混合检索（可选） | 监控与失败重试、索引延迟指标 |
| **P3** | CI：断链/死 slug 检查；Preview 环境；生产发布 checklist | 流水线与运维 Runbook |

---

## 9. 风险与对策（摘要）

| 风险 | 对策 |
|------|------|
| MDX 运行时与任意组件 | 严格 MDX 组件白名单；禁止任意 JSX 执行未审核代码。 |
| FTS 与向量不同步 | 统一「发布」事件驱动；版本号校验。 |
| SQLite FTS 瓶颈 | 提前定义迁移到 Meilisearch 的阈值（QPS、库大小）。 |
