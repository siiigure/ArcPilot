### 一、核心产品设计

- 概念统一：
    
    - 所有原本的创建空间功能，改为 “协作空间”。
    - 示例：用户 A 创建“苏州古城保护调研”协作空间，邀请用户 B，一起上传 PDF 规范、GIS 截图、Python 脚本等，仅成员可见。
- 角色与权限（示例，可按需调整）：
    
    - Owner：创建人，拥有所有权限（上传、删除、邀请、修改配置/配额）。
    - Editor：上传、删除（至少能删自己上传的）、查看所有资源，不能调配额/改 Owner。
    - Viewer：仅查看/下载资源，无上传、删除、邀请等操作。

### 二、数据模型设计（后端）

#### 1. 协作空间表 `CollabSpace`

- 字段建议：
    - `id`: 主键
    - `name`: 协作空间名称（如“苏州古城保护调研”）
    - `description`: 说明（可选）
    - `owner_id`: 外键 -> `User.id`
    - `storage_quota`: 空间配额（单位 Byte 或 MB）
    - `used_storage`: 已用存储空间
    - `created_at`, `updated_at`
- 作用：
    - 表示一个独立的协作空间实例，承载配额、所有者信息。

#### 2. 成员表 `CollabSpaceMember`

- 字段建议：
    - `id`: 主键
    - `space_id`: 外键 -> `CollabSpace.id`
    - `user_id`: 外键 -> `User.id`
    - `role`: `"owner" | "editor" | "viewer"`
    - `joined_at`
- 作用：
    - 记录每个空间的成员与角色，用于权限控制。

#### 3. 资源表 `SpaceAsset`（文件 / 脚本等）

- 字段建议：
    - `id`: 主键
    - `space_id`: 外键 -> `CollabSpace.id`
    - `uploader_id`: 外键 -> `User.id`
    - `logical_name`: 用户看到的文件名（如 `南京路更新.shp`）
    - `version`: 版本号（int，从 1 递增，允许同名多版本）
    - `type`: `"pdf" | "image" | "gis" | "python" | "other"`（可据 MIME 归类）
    - `storage_path` / `url`: 实际存储路径或对象存储 URL
    - `size`: 文件大小（Byte，用于配额计算）
    - `is_deleted`: 是否已删除（软删除）
    - `deleted_at`, `deleted_by_user_id`（可选）
    - `created_at`
- 逻辑：
    - 同名版本控制：
        - 通过 `(space_id, logical_name, version)` 唯一约束。
        - 上传时：查找该空间该 `logical_name` 的 `max(version)`，新版本 = `max + 1`。
        - 不报错、不覆盖，自动升级版本。
    - 删除策略：
        - 应用层执行软删除（`is_deleted = True`），并触发文件物理删除（立即或定期任务）。
        - 删除成功后需扣减 `CollabSpace.used_storage`。

#### 4. 邀请表 `SpaceInvite`（支持未注册用户）

- 字段建议：
    - `id`: 主键
    - `space_id`: 外键 -> `CollabSpace.id`
    - `email`: 被邀请邮箱
    - `token`: 邀请令牌（随机串）
    - `expires_at`
    - `inviter_id`: 邀请人 `User.id`
    - `status`: `"pending" | "accepted" | "expired | cancelled"`
    - `created_at`, `updated_at`
- 作用：
    - 支持未注册邮箱的“邀请注册 + 自动入空间”闭环。
    - 如有 Redis，可额外用 `invite:{token}` 做缓存，但核心状态仍以数据库为准。

### 三、权限与访问控制（后端）

#### 1. 通用权限检查

- 编写依赖/工具函数，例如：
    
    - `get_current_space_member(session, current_user, space_id)`：
        - 查询 `CollabSpaceMember`，若无记录则抛 `HTTP 403` 或 `404`。
        - 返回成员信息（包含 `role`）作为后续权限判断依据。
- 所有空间相关接口必须在业务逻辑前调用该函数，包括：
    
    - 获取空间详情
    - 列出空间成员
    - 列出/上传/删除资源
    - 邀请成员等

#### 2. 角色能力矩阵（推荐）

- Owner：
    
    - 查看/下载空间资源
    - 上传资源
    - 删除任意资源
    - 邀请/移除成员，修改角色
    - 修改空间资料与配额
- Editor：
    
    - 查看/下载资源
    - 上传资源
    - 删除自己上传的资源（或可配置是否允许删他人）
- Viewer：
    
    - 只能查看/下载资源
    - 无上传 / 删除 / 邀请 / 修改配额等权限
- 后端在各接口中根据 `role` 做严格校验，不能只靠前端隐藏按钮。

### 四、后端接口设计（REST）

#### 1. 协作空间管理

- `POST /api/v1/spaces`
    
    - 功能：创建协作空间。
    - 请求体：`{ name, description?, storage_quota? }`
    - 逻辑：
        - 创建 `CollabSpace`，`owner_id = current_user.id`，`used_storage = 0`，配额可有默认值。
        - 在 `CollabSpaceMember` 中插入一条记录：角色 `owner`。
- `GET /api/v1/spaces`
    
    - 功能：列出当前用户参与的所有协作空间。
    - 基于 `CollabSpaceMember` 关联查询。
- `GET /api/v1/spaces/{space_id}`
    
    - 功能：获取空间详情。
    - 返回字段中包含当前用户角色 `current_user_role`，供前端做 UI 控制。

#### 2. 成员 & 邀请

- `GET /api/v1/spaces/{space_id}/members`
    
    - 功能：空间成员列表。
    - 权限：`owner` / `editor` / `viewer` 均可访问（具体看产品需求）。
- `POST /api/v1/spaces/{space_id}/invite`
    
    - 功能：邀请成员加入空间。
    - 请求体示例：`{ email: string, role: "editor" | "viewer" }`
    - 权限：通常仅 `owner` 可邀请（或允许 `editor`，视需求）。
    - 逻辑：
        1. 检查当前用户对该空间是否有邀请权限。
        2. 查找 `User` 表看该邮箱是否已注册。
            - 已注册：
                - 若尚未是成员，创建 `CollabSpaceMember`；
                - 发送“你被邀请加入 XX 协作空间”的邮件通知。
            - 未注册：
                - 生成 `SpaceInvite` 记录，包含 `token` 和 `expires_at`；
                - 可同时写入 Redis 作为辅助缓存。
                - 发送邀请邮件，附上链接 `https://frontend/signup?invite_token=XXX`。
- `POST /api/v1/invites/accept`
    
    - 功能：用户注册/登录后接受邀请。
    - 请求体：`{ invite_token }`
    - 逻辑：
        1. 解析 token，验证未过期、状态为 `pending`。
        2. 获取当前登录用户的 `email`，需与邀请记录邮箱匹配。
        3. 创建 `CollabSpaceMember` 成员记录；更新 `SpaceInvite.status = "accepted"`。

### 五、资源上传 / 版本 / 配额 / 删除

#### 1. 空间资源上传（版本 + 配额）

- 推荐设计两类接口（取决于存储方式）：

##### 方案 A：对象存储 + 预签名 URL（推荐）

1. `POST /api/v1/spaces/{space_id}/assets/presign`
    
    - 参数：`{ logical_name, mime_type, size }`
    - 流程：
        - 权限检查：需 `owner` / `editor`。
        - 检查 `CollabSpace.used_storage + size <= storage_quota`，不足则报错（如 413 + 业务码）。
        - 计算本次上传对应的版本号：
            - 查 `SpaceAsset` 中该 `space_id + logical_name` 的最大 `version`，新版本 = `max + 1`。
        - 生成对象存储 key（例如 `spaces/{space_id}/{uuid}/v{version}-{fileName}`）。
        - 调用对象存储 SDK 生成上传的 presigned URL。
        - 返回：`{ upload_url, asset_meta: { logical_name, version, storage_key, size } }`。
2. 前端使用 `upload_url` 直接 PUT/POST 文件到对象存储，可利用 SDK 做分片与断点续传。
    
3. `POST /api/v1/spaces/{space_id}/assets/complete`
    
    - 参数：`{ logical_name, version, storage_key, size, type? }`
    - 流程：
        - 再次校验配额（防止恶意绕过预检）。
        - 写入 `SpaceAsset` 记录。
        - 在事务中增加 `CollabSpace.used_storage += size`。

##### 方案 B：本机存储 + 分片上传（如果不使用对象存储）

- 核心接口：
    - `POST /spaces/{space_id}/assets/init-upload` -> 返回 `upload_id` + 版本信息。
    - `POST /spaces/{space_id}/assets/upload-chunk?upload_id=...&index=...` 分片上传。
    - `POST /spaces/{space_id}/assets/complete-upload` 合并分片，写数据库并更新配额。

> 两种方案思想一致：先配额检查 -> 分片/直传 -> 完成后写数据库并更新配额。

#### 2. 列出 & 查看资源

- `GET /api/v1/spaces/{space_id}/assets`
    
    - 功能：列出该空间资源。
    - 默认只返回每个 `logical_name` 的最新版本；可加参数 `?all_versions=true` 来返回全部版本。
    - 返回字段中包含：`logical_name`, `version`, `type`, `size`, `uploader`, `created_at` 等。
- `GET /api/v1/spaces/{space_id}/assets/{asset_id}`
    
    - 功能：下载/预览具体资源。
    - 可返回：
        - 若用对象存储：转发或生成下载用 presigned URL。
        - 若本机存储：直接以流形式返回文件内容。

#### 3. 删除资源（含磁盘文件处理）

- `DELETE /api/v1/spaces/{space_id}/assets/{asset_id}`
    - 权限：
        - `owner` 可以删除任意资源；
        - `editor` 仅可删除自己上传的资源（推荐）；
        - `viewer` 无删除权限。
    - 逻辑：
        1. 校验空间成员和角色。
        2. 软删除：更新 `SpaceAsset.is_deleted = True`，记录 `deleted_at`, `deleted_by_user_id`。
        3. 物理删除：
            - 简化方案：同步删除对象存储/磁盘中的文件；
            - 稍复杂方案：写一条 background job，异步真正删除，避免阻塞请求。
        4. 更新 `CollabSpace.used_storage -= asset.size`（注意下限为 0）。

### 六、前端页面与交互设计

#### 1. 导航与路由结构

- 在左侧导航 `LeftSidebar` 中添加主入口：“协作空间”。
- 路由建议（基于现有 `_layout` 结构）：
    - `/_layout/spaces`：协作空间列表页。
    - `/_layout/spaces/$spaceId`：协作空间详情页。

#### 2. 协作空间列表页（`/_layout/spaces`）

- 显示：
    
    - 当前用户参与的空间列表（调用 `GET /spaces`）。
    - 每个空间卡片展示名称、简介、创建者、创建时间等。
- 操作：
    
    - 顶部 `“创建协作空间”` 按钮（替代旧文案“创建空间”）：
        - 点击弹出对话框表单：`name`, `description?`, 可选 `storage_quota`。
        - 提交调用 `POST /spaces`；成功后刷新列表。

#### 3. 协作空间详情页（`/_layout/spaces/$spaceId`）

- 基础信息：
    
    - 顶部展示空间名称、简介、配额使用情况（如 `已用 2.3GB / 5GB`）。
- Tabs 结构建议：
    
    - `资源（Assets）` tab：
        - 表格 / 卡片列出资源：文件名、最新版本号、类型、大小、上传者、上传时间。
        - 每行操作：查看历史版本、下载。
        - 顶部“上传资源”按钮：
            - 上传组件支持大文件进度展示、重试、取消；
            - 实际调用 `presign` -> 浏览器直传 -> `complete`。
    - `成员（Members）` tab：
        - 列出成员及其角色。
        - 若当前用户为 `owner`：
            - 显示“邀请成员”输入框（邮箱 + 角色选择），调用 `POST /spaces/{spaceId}/invite`。
            - 可增加更改成员角色/移除成员操作。
    - 其他（可选）：`设置` tab 用于修改空间名称、说明、配额等，仅 `owner` 可见。

#### 4. 前端权限控制（按钮显隐）

- 空间详情接口返回 `current_user_role`。
- 页面中根据角色条件渲染：
    - Viewer：
        - 只能看到资源列表和下载按钮；
        - 不显示“上传资源”、“删除”、“邀请成员”、“修改空间设置”等按钮。
    - Editor：
        - 显示“上传资源”；
        - 对自己的资源行显示“删除”按钮；
        - 默认不显示“邀请成员”、“配额设置”。
    - Owner：
        - 显示全部管理功能：上传、删除、邀请、成员管理、修改配额、删除空间等。

> 这样保证：即使 viewer 打开详情页，也看不到不该有的操作入口；后端再做二次验证，避免越权。

### 七、上传体验与容错

- 断点续传 & 进度条：
    - 前端使用分片/直传 SDK（取决于存储），配合 `presign` 或分片接口。
    - 显示上传进度条、速度、剩余时间；网络断开自动重试，或允许用户手动重试。
- 前端不“卡死”：
    - 上传在后台异步执行；主界面保持可用。
    - 上传列表/任务栏展示当前上传任务状态（上传中、完成、失败）。
    - 上传完成后，自动刷新资源列表。

### 八、整体闭环回顾

- 协作空间生命周期：
    1. A 在前端点击“创建协作空间”，后端创建 `CollabSpace` + `CollabSpaceMember(owner)`。
    2. A 邀请 B：若 B 已注册，直接加入；若未注册，后端生成 `SpaceInvite` + 邀请令牌发送邮件。
    3. B 通过邀请链接注册/登录，调用 `accept` 接口后自动成为空间成员。
    4. A、B 在空间内上传 PDF、GIS 截图、Python 脚本等，带版本号管理，只有成员能访问。
    5. 系统通过 `storage_quota + used_storage` 控制资源占用，防止恶意/误操作导致存储爆炸。
    6. 支持资源删除（软删 + 实体文件移除）、版本回溯、大文件上传与断点续传。
    7. 前后端基于 `role` 做双层权限控制，确保 Viewer 只能看不能改。

  - `GET /api/v1/spaces/{space_id}`：空间详情（`space_id` 为 `public_id` UUID）；非成员返回 404，避免枚举空间是否存在。
  - `GET /api/v1/spaces/{space_id}/members`：成员列表。
  - `POST /api/v1/spaces/{space_id}/invite`：仅 owner 可调用，生成邀请码（可选 `email` 仅占位），`role` 仅允许 `editor`/`viewer`。
  - `POST /api/v1/invites/accept`：请求体 `{ invite_code }`，登录用户接受邀请并入组。

#### 9.4 阶段四：方案 A 资源上传（本机模拟，已完成）

- **设计**
  - `POST /api/v1/spaces/{space_id}/assets/presign`：校验成员角色（owner/editor）、配额与版本号，签发短期 JWT，返回 `upload_url`（`PUT`）、`upload_token` 与 `asset_meta`。
  - `PUT /api/v1/spaces/assets/upload/{token}`：不依赖登录态，仅凭 JWT 写入本机目录（根目录由 `SPACE_ASSETS_LOCAL_ROOT` 配置，默认 `data/space_assets`），用于模拟对象存储直传。
  - `POST /api/v1/spaces/{space_id}/assets/complete`：请求体 `{ upload_token }`，校验当前用户与 JWT 中 `uploader_id` 一致后写入 `SpaceAsset` 并增加 `CollabSpace.used_storage`。
  - `GET /api/v1/spaces/{space_id}/assets`：支持 `all_versions` 查询参数；默认仅返回每个 `logical_name` 的最新版本。
  - `GET /api/v1/spaces/{space_id}/assets/{asset_id}`：`FileResponse` 下载。
  - `DELETE /api/v1/spaces/{space_id}/assets/{asset_id}`：软删 + 尝试物理删文件 + 扣减配额；权限使用 `can_delete_asset`（owner 可删任意，editor 仅自己上传）。
- **实现文件**
  - `app/core/space_asset_upload.py`：JWT 载荷、路径安全、`mime` 粗分类。
  - `app/api/routes/space_assets.py`：上述路由；`app/api/main.py` 中 `include_router(space_assets.router)`。
  - `app/core/config.py`：新增 `SPACE_ASSETS_LOCAL_ROOT`。
- **测试与夹具**
  - `tests/api/routes/test_space_assets.py`：预签名后手动写入字节再 `complete` 的闭环。
  - `tests/conftest.py`：session 结束时补充删除 `SpaceAsset`、`SpaceInvite`、`CollabSpaceMember`、`CollabSpace`，避免外键阻塞清理测试用户。
- **环境与测试说明**
  - 全量 pytest 依赖可用的 PostgreSQL（`init_db` 建表）；若数据库不可达，会话级测试可能长时间阻塞，请在本地确认 `.env` 中库地址后再跑 `uv run pytest ...`。
