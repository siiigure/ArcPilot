## 2026-03-31：5173 页面加载失败 + 发帖一直加载后重置

### 一、现象
- 前端 `http://localhost:5173` 频繁出现“加载失败，请稍后重试”。
- 点击“发帖”后，按钮长时间显示“发布中…”，随后界面回到初始空白状态，未成功发布。

### 二、根因
- 后端服务表面上已启动，但实际卡在 FastAPI 的 `startup` 阶段，未真正就绪。
- 卡点位于 `backend/app/core/migrate_plan_b.py` 的迁移 SQL（`ALTER TABLE ...`）。
- 数据库中存在会话持锁/锁等待，导致迁移语句阻塞；阻塞链会影响后续查询与写入请求。
- 前端请求后端超时或失败，因此触发“加载失败”；发帖请求未成功，表现为持续 loading 和流程回退。

### 三、处理过程
1. 通过健康检查确认后端不可用：`/api/v1/utils/health-check/` 超时。
2. 检查后端启动日志，定位到 `Waiting for application startup` 长时间停滞。
3. 拆分执行初始化流程，确认卡在迁移 SQL 阶段。
4. 检查 PostgreSQL 活跃会话，发现多个 `ALTER TABLE` 锁等待与 `idle in transaction`。
5. 修改迁移逻辑：
   - 每条迁移 SQL 单独事务执行；
   - 设置 `lock_timeout` 与 `statement_timeout`；
   - 单条语句失败时回滚并记录 warning，避免整体 startup 卡死。
6. 清理已堆积的阻塞会话，恢复数据库可用性。
7. 再次验证健康检查，后端恢复 `200`，应用完成启动。

### 四、如何避免（预防措施）
- **迁移策略**：启动期迁移必须“可失败、可跳过、不可无限阻塞”。
- **超时保护**：涉及 DDL 的逻辑必须设置锁等待超时与语句执行超时。
- **事务粒度**：不要把整批迁移放进一个大事务；单条失败不应污染后续执行。
- **会话治理**：定期检查并清理 `idle in transaction` 会话，减少锁链风险。
- **就绪探针**：开发环境也应先看健康检查是否就绪，再判断前端问题。

### 五、遇到同类问题的标准排查步骤
1. 先看后端健康检查是否可达（不要先怀疑前端）。
2. 看后端是否卡在 startup（尤其是初始化/迁移/建表逻辑）。
3. 查数据库锁与活跃会话（是否有 DDL 等锁、`idle in transaction`）。
4. 如有锁链，先解除阻塞，再重启后端验证。
5. 补上超时、回滚与降级策略，避免再次卡死。

## 2026-04-01：8080 无法访问 + 5173 登录失败

### 一、现象
- 浏览器无法打开 `http://localhost:8080`（连接被拒绝、超时或无法建立连接）。
- `http://localhost:5173` 无法正常使用，或页面能开但**登录失败**（接口报错、一直转圈、提示凭证无效等）。

### 二、根因
- **8080**：本仓库中 **8080 并非后端 API 默认端口**。`compose.yml` / `compose.override.yml` 将 **Adminer**（数据库 Web 管理）或 Traefik 相关服务映射到 **8080**；若未启动对应 **Docker Compose** 服务，本机**无进程监听 8080** 属预期情况，易被误认为「应用挂了」。
- **API 端口**：后端 OpenAPI 与 `frontend/.env` 中 `VITE_API_URL` 约定为 **`http://localhost:8000`**；前端开发服务器为 **5173**。
- **5173 登录失败**：常见为（1）**后端 8000 未启动或健康检查未通过**，前端请求登录/鉴权失败；（2）**换库、清库、删用户后仍使用旧 `access_token`**，鉴权拒绝（参见此前对 `deps` 的调整：应清 token 重登）；（3）**5173 未启动**（未执行 `npm run dev` / `pnpm dev`），页面本身不可达。与 **2026-03-31** 条目中「后端未就绪导致前端表现异常」的排查主线一致。

### 三、处理过程
1. 确认 **API 是否存活**：访问 `http://localhost:8000/api/v1/utils/health-check/`，应返回 **200**（不要先用 8080 判断后端）。
2. 需要 **5173**：在 `arcpilot/frontend` 执行 `npm run dev`（或项目文档中的等价命令），确认终端提示监听 `5173`。
3. 需要 **8080（Adminer）**：在 `arcpilot` 目录按 `development.md` / Compose 文档启动 **docker compose**，确认 `8080:8080` 映射的服务已 `Up`。
4. **登录失败**：检查浏览器 Network 中登录与后续请求的状态码；清除 `localStorage` 中的 `access_token` 后重新注册/登录；确认 `VITE_API_URL` 指向当前运行的后端地址与端口（默认 **8000**）。
5. 若 **8000 健康检查卡住或超时**：按 **2026-03-31** 条目第三节、第五节排查 startup、迁移锁与数据库会话。

### 四、如何避免（预防措施）
- 在团队文档中固定写清：**前端 5173、后端 API 8000、Adminer（若用 Docker）8080**，避免端口混淆。
- 数据库重建或迁移后，提示成员**重新登录**或清理本地 token。
- 本地开发可先脚本化探测 `health-check` 再打开前端页面。

### 五、遇到同类问题的标准排查步骤
1. `GET /api/v1/utils/health-check/`（8000）是否 200。
2. `5173` 是否有进程监听（如 `lsof -i :5173` 或开发服务器终端输出）。
3. 若仅 **8080** 打不开：是否本就需要 **Docker** 起的 Adminer；若不需要，可忽略 8080。
4. 登录失败：看 Network 响应 `detail`，清 token、对齐 `VITE_API_URL`、确认后端日志无 500。
