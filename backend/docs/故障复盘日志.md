## 2026-03-31：5173 页面加载失败 + 发帖一直加载后重置

### 一、现象
- 前端 `http://localhost:5173` 频繁出现“加载失败，请稍后重试”。
- 点击“发帖”后，按钮长时间显示“发布中…”，随后界面回到初始空白状态，未成功发布。

### 二、根因
- 后端服务表面上已启动，但实际卡在 FastAPI 的 `startup` 阶段，未真正就绪。
- 卡点位于 `backend/app/core/migrate_plan_b.py` 的迁移 SQL（`ALTER TABLE ...`）。
- 数据库中存在会话持锁/锁等待，导致迁移语句阻塞；阻塞链会影响后续查询与写入请求。
- 前端请求后端超时或失败，因此触发“加载失败”；发帖请求未成功，表现为持续 loading 和流程回退。

### 三、处理过程
1. 通过健康检查确认后端不可用：`/api/v1/utils/health-check/` 超时。
2. 检查后端启动日志，定位到 `Waiting for application startup` 长时间停滞。
3. 拆分执行初始化流程，确认卡在迁移 SQL 阶段。
4. 检查 PostgreSQL 活跃会话，发现多个 `ALTER TABLE` 锁等待与 `idle in transaction`。
5. 修改迁移逻辑：
   - 每条迁移 SQL 单独事务执行；
   - 设置 `lock_timeout` 与 `statement_timeout`；
   - 单条语句失败时回滚并记录 warning，避免整体 startup 卡死。
6. 清理已堆积的阻塞会话，恢复数据库可用性。
7. 再次验证健康检查，后端恢复 `200`，应用完成启动。

### 四、如何避免（预防措施）
- **迁移策略**：启动期迁移必须“可失败、可跳过、不可无限阻塞”。
- **超时保护**：涉及 DDL 的逻辑必须设置锁等待超时与语句执行超时。
- **事务粒度**：不要把整批迁移放进一个大事务；单条失败不应污染后续执行。
- **会话治理**：定期检查并清理 `idle in transaction` 会话，减少锁链风险。
- **就绪探针**：开发环境也应先看健康检查是否就绪，再判断前端问题。

### 五、遇到同类问题的标准排查步骤
1. 先看后端健康检查是否可达（不要先怀疑前端）。
2. 看后端是否卡在 startup（尤其是初始化/迁移/建表逻辑）。
3. 查数据库锁与活跃会话（是否有 DDL 等锁、`idle in transaction`）。
4. 如有锁链，先解除阻塞，再重启后端验证。
5. 补上超时、回滚与降级策略，避免再次卡死。
